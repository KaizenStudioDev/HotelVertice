1. RESUMEN EJECUTIVO
================================================================================

El proyecto Hotel Vertice es un sistema web full-stack de gestion de reservas
hoteleras desarrollado con Node.js/Express + Supabase (backend) y React/Vite/
TypeScript + Tailwind CSS (frontend). El equipo ha completado los Sprints 1, 2
y 3, y se encuentra ejecutando el Sprint 4 (pruebas y optimizaciones).

La arquitectura general es limpia, profesional y escalable. El flujo core de
reservas (registro, login, ver habitaciones, reservar, confirmar, ver perfil)
esta operativo de extremo a extremo. Sin embargo, existen bugs criticos de
logica, caracteristicas faltantes del enunciado del profesor, y areas de calidad
de codigo que deben atenderse para alcanzar la maxima calificacion.

ESTADO POR SPRINT:

2. EVALUACION DE REQUISITOS FUNCIONALES (segun enunciado del profesor)
================================================================================

RF-01  Registro y login de usuarios (huespedes, recepcionistas, admins)
       Estado:   PARCIALMENTE CUMPLIDO
       Lo hecho: Login y registro de usuarios funcional con JWT y Supabase Auth.
                 Roles definidos en el sistema (guest, admin).
       Faltante: Rol "recepcionista" no esta implementado ni diferenciado del
                 rol "guest". No hay flujo de login separado para admin/recep.
       Evidencia: backend/src/controllers/authController.ts,
                  frontend/src/pages/Login.tsx

RF-02  Busqueda y reserva de habitaciones con disponibilidad en tiempo real
       Estado:   PARCIALMENTE CUMPLIDO
       Lo hecho: Lista de habitaciones con filtros de piso y vista. Reserva con
                 validacion de traslapes en el backend.
       Faltante: El buscador de la landing page (fechas, num. huespedes) NO
                 pasa los datos al contexto de reserva. La disponibilidad
                 por fechas no se filtra en la vista de habitaciones.
       Evidencia: frontend/src/pages/Landing.tsx, frontend/src/pages/Rooms.tsx,
                  frontend/src/context/BookingContext.tsx

RF-03  Gestion de reservas: confirmacion, modificacion, cancelacion
       Estado:   PARCIALMENTE CUMPLIDO
       Lo hecho: Confirmacion al crear reserva (status='confirmed').
                 Cancelacion funcional desde el backend.
       Faltante: Modificacion de reservas NO implementada (ni frontend ni
                 backend). La cancelacion desde el perfil de usuario no es
                 visible en el frontend (solo el admin puede ver reservas
                 en el dashboard, pero no cancelarlas desde la UI del usuario).
       Evidencia: backend/src/controllers/reservationController.ts (linea 82),
                  frontend/src/pages/UserProfile.tsx

RF-04  Check-in / Check-out y facturacion
       Estado:   NO CUMPLIDO
       Lo hecho: Generacion de PDF como comprobante de reserva (jsPDF).
       Faltante: No existen endpoints ni UI para registrar check-in o check-out.
                 No hay facturacion formal. El PDF generado es un "comprobante",
                 no una factura con impuestos, numero de factura, etc.
       Evidencia: frontend/src/services/pdfService.ts

RF-05  Panel administrativo: reportes de ocupacion, ingresos, lista de huespedes
       Estado:   MINIMO CUMPLIDO
       Lo hecho: AdminDashboard con tres tabs: estadisticas generales (numeros
                 hardcodeados), lista de reservas (datos reales desde API),
                 y estado de habitaciones.
       Faltante: - Reportes de ocupacion diaria/semanal: NO IMPLEMENTADOS.
                 - Reportes de ingresos por periodo: NO IMPLEMENTADOS.
                 - Lista de huespedes descargable: NO IMPLEMENTADA.
                 - El tab "Reservas" del admin muestra texto de placeholder:
                   "Interfaz de busqueda avanzada... (Proxima actualizacion)".
                 - No hay CRUD de habitaciones desde el dashboard.
       Evidencia: frontend/src/pages/AdminDashboard.tsx (lineas con placeholder)

RF-06  Notificaciones automaticas (email/SMS)
       Estado:   NO CUMPLIDO
       Lo hecho: Nada implementado.
       Faltante: Integracion con API externa de email (SendGrid, Resend, etc.)
                 o SMS. No hay ninguna referencia en el backend.

RF-07  Integracion con calendario (FullCalendar, estados de habitaciones)
       Estado:   NO CUMPLIDO
       Lo hecho: Nada implementado.
       Faltante: El Sprint 3 exige un calendario interactivo con FullCalendar
                 que muestre estados (libre, reservada, ocupada, mantenimiento).
                 No existe ni la dependencia instalada.
       Evidencia: frontend/package.json (FullCalendar ausente)

RNF-01 Interfaz responsive y accesible (WCAG 2.1)
       Estado:   PARCIALMENTE CUMPLIDO

RNF-02 Seguridad: JWT, prevencion SQLi/XSS, encriptacion PII
       Estado:   PARCIALMENTE CUMPLIDO
       Lo hecho: JWT implementado correctamente con Supabase Auth.
                 Supabase maneja parametrizacion de queries (previene SQLi).
       Faltante: - El archivo .env con credenciales esta siendo trackeado por
                 Git. NOTA: El repositorio es privado y de acceso restringido

                 recomendacion de buenas practicas para cuando el proyecto
                 se despliegue o el repo cambie de visibilidad.
                 - No hay rate limiting en endpoints de autenticacion
                   (vulnerable a fuerza bruta).
                 - No hay sanitizacion de inputs en el frontend antes de
                   enviar a la API.
                 - Los mensajes de error del backend exponen detalles internos
                   directamente (error.message sin sanitizar).

       Estado:   PENDIENTE

       Faltante: No se ha realizado ninguna configuracion de despliegue.
                 No existe railway.json, vercel.json, ni Dockerfile.

RNF-04 Control de versiones Git, codigo modular, pruebas >70% cobertura, UML

       Lo hecho: Repositorio Git activo. Cobertura de pruebas del backend

       Faltante: Solo hay 1 commit en el historial ("Initial commit"), lo que

RNF-05 Rendimiento: soporte 100 reservas concurrentes simuladas
       Estado:   NO EVALUADO
       Lo hecho: Nada.
       Faltante: Pruebas de carga no implementadas. Herramientas como k6 o

4. EVALUACION DEL BACKEND
================================================================================

4.1 ESTRUCTURA Y ARQUITECTURA
  + Express server correctamente configurado con rutas modularizadas.
  + Separacion clara: controllers/, routes/, middlewares/, utils/.
  - Directorios models/ y services/ estan VACIOS. Se planearon pero nunca
    se usaron — la logica de negocio esta mezclada en los controllers.

4.2 CONTROLLERS

  authController.ts (56 lineas)
  + Registro, login y getProfile implementados correctamente.
  + Manejo de errores con status HTTP apropiados.
  - Uso de "error: any" en catch blocks (mala practica en TypeScript).
  - getProfile() asume que existe una tabla "profiles" pero no esta
    documentado que esta tabla exista o tenga la estructura correcta.

  reservationController.ts (116 lineas)
  + Validacion de fechas (check-in < check-out). CORRECTO.
  + Deteccion de traslapes con query de overlapping. CORRECTO.
  + Calculo de precio (noches * base_price). CORRECTO.
  - LINEA 8: Query join incorrecto:
      .select('*, rooms(*), room_types:rooms(room_types(*))')
    El alias "room_types:rooms(...)" es confuso y probablemente incorrecto.
    Deberia ser: .select('*, rooms(*, room_types(*))')
  - No hay validacion del tipo de room_id (podria enviarse un string).
  - No hay endpoint para MODIFICAR una reserva existente.

  roomController.ts (40 lineas)
  + GET all rooms y GET by ID funcionan correctamente.
  - Faltan endpoints de administracion: POST (crear), PUT/PATCH (editar),
    DELETE (eliminar) habitacion — necesarios para el panel admin.

4.3 MIDDLEWARES

  authMiddleware.ts (53 lineas)
  + authenticate(): Validacion de JWT via Supabase. CORRECTO.
  + authorize(): RBAC implementado correctamente.
  - CRITICO: La funcion authorize() NUNCA se usa en ningun route.
    Las rutas de admin no estan protegidas por rol.
    Cualquier usuario autenticado puede acceder a endpoints de admin.

4.4 RUTAS
  + authRoutes.ts, roomRoutes.ts, reservationRoutes.ts bien estructuradas.
  - Todos los GET de habitaciones son publicos — correcto.
  - Las rutas de reservas estan autenticadas — correcto.
  - No existen rutas de administracion (/api/admin/*).

4.5 PRUEBAS DEL BACKEND
  + 5 archivos de prueba: auth, rooms, reservations, middleware, full_qa.
  + Cobertura ~76% (cumple requisito >70%).
  + Uso de Vitest + Supertest (eleccion correcta).
  - auth.test.ts linea 26: Acepta tanto 201 como 400 — demasiado permisivo.
  - full_qa.test.ts linea 30: Usa UUID ficticio que probablemente no existe
    en la DB de pruebas — el test puede pasar por razones incorrectas.
  - reservations.test.ts: vi.mock() despues de beforeAll puede causar
    comportamiento no determinista.

4.6 SEGURIDAD
  CRITICO: El archivo .env esta commiteado en el repositorio (visible en
           git history). Contiene SUPABASE_SERVICE_ROLE_KEY que da acceso
           total a la base de datos.
  - No existe .env.example para documentar variables requeridas.
  - No hay rate limiting en /api/auth/login (riesgo de fuerza bruta).
  - Los mensajes de error devuelven error.message del sistema directamente.

5.2 PAGINAS IMPLEMENTADAS (11/10 planeadas + 1 extra)

  + Features.tsx     — Servicios del hotel, spa, gimnasio.
  + Rooms.tsx        — Lista de habitaciones con filtros, integrada a la API.
  + BookingDetails.tsx — Formulario de datos del huesped.
  + Payment.tsx      — Formulario de pago (mock).
  + PaymentConfirmation.tsx — Confirmacion con descarga de PDF.
  + Login.tsx / Register.tsx — Autenticacion funcional.
  + UserProfile.tsx  — 3 tabs: mis reservas, datos personales, seguridad.
  + AdminDashboard.tsx — Panel administrativo basico.
  + UIKitTest.tsx    — Pagina de prueba del design system (util para dev).

5.3 BUGS CRITICOS EN EL FRONTEND

  BUG-F01 [CRITICO] Payment.tsx linea 16:
    const totalDays = 5; // HARDCODEADO
    El numero de noches no se calcula a partir de las fechas reales del
    BookingContext. El precio total siempre sera incorrecto.

  BUG-F02 [CRITICO] PaymentConfirmation.tsx linea ~27:
    (booking.pricePerNight || 0) * 2
    Multiplica por 2 en lugar del numero real de noches. El comprobante
    PDF mostrara siempre el precio equivocado.

  BUG-F03 [CRITICO] BookingDetails.tsx linea ~97-98:
    Igual que BUG-F01, calcula precio con noches fijas en lugar de las
    fechas del contexto.

  BUG-F04 [ALTO] Landing.tsx — Buscador no funcional:
    El formulario de busqueda (fechas, huespedes, tipo) NO actualiza el
    BookingContext. Al navegar a /rooms, no se aplican los criterios
    de busqueda ingresados.

  BUG-F05 [ALTO] Rooms.tsx — Disponibilidad no verificada por fechas:
    La pagina filtra habitaciones por status === 'available' pero no
    verifica contra las fechas seleccionadas por el usuario. Una habitacion
    puede mostrarse como disponible aunque tenga una reserva en esas fechas.

  BUG-F06 [MEDIO] Navbar.tsx — Menu mobile no implementado:
    El boton hamburguesa esta en el HTML pero no tiene handler. En mobile,
    el menu de navegacion es completamente inaccesible.

  BUG-F07 [MEDIO] UserProfile.tsx — Tabs de edicion sin funcionalidad:
    Los formularios de "Datos Personales" y "Seguridad" hacen preventDefault()
    pero no llaman a ningun servicio de la API. Los cambios no se guardan.

  BUG-F08 [MEDIO] AdminDashboard.tsx — Tab "Reservas" es placeholder:
    El segundo tab del dashboard admin muestra el texto:
    "Interfaz de busqueda avanzada... (Proxima actualizacion)"
    No tiene ninguna funcionalidad real.

  BUG-F09 [BAJO] frontend/src/lib/supabaseClient.ts — Importado pero inutil:
    Este archivo crea un cliente de Supabase directamente en el frontend,
    pero nunca se usa (toda la comunicacion va por la API del backend).
    Codigo muerto que genera confusion.

  BUG-F10 [BAJO] BookingContext.tsx — Fechas hardcodeadas como estado inicial:
    checkIn: '2026-04-15', checkOut: '2026-04-20'
    Las fechas iniciales son fechas fijas pasadas, no fechas dinamicas.

CONCLUSION
================================================================================

El proyecto tiene una base tecnica solida y un diseno visual de calidad
profesional. El flujo principal de reservas es funcional de extremo a extremo,
lo cual es el logro mas importante. Sin embargo, para cumplir al 100% con los
requisitos del enunciado del profesor, se deben atacar urgentemente los bugs
criticos de calculo de precio (que el profesor verificara en la demo), la
implementacion del calendario FullCalendar (requisito explicito del Sprint 3),
y el historial de commits (criterio de colaboracion del 15%).

El calendario interactivo y las notificaciones de email son los dos gaps mas
grandes frente a los requisitos del enunciado. Las demas correcciones son de
calidad de codigo y se pueden abordar sistematicamente.
