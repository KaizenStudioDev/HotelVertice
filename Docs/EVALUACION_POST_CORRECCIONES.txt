ETAPA 1 — BUGS CRITICOS DE PRECIO [COMPLETADA]
  Problema: El precio total en todo el flujo de reserva estaba mal.
  - Payment.tsx tenia const totalDays = 5 hardcodeado.
  - PaymentConfirmation.tsx multiplicaba precio * 2 fijo.
  - BookingDetails.tsx tambien usaba 5 noches fijas.
  - Fechas iniciales del BookingContext eran '2026-04-15' y '2026-04-20' fijas.

  Solucion aplicada:
  - Se creo y exporto calcNights(checkIn, checkOut) en BookingContext.tsx.
    Calcula: Math.max(1, Math.ceil((checkOut - checkIn) / 86400000))
  - Todos los componentes ahora usan calcNights() con las fechas reales.
  - Payment.tsx guarda totalPrice en el contexto antes de navegar a confirmacion.
  - PaymentConfirmation usa booking.totalPrice (guardado por Payment).
  - Fechas iniciales son dinamicas: new Date() y tomorrow.

  Archivos modificados:
    frontend/src/context/BookingContext.tsx
    frontend/src/pages/Payment.tsx
    frontend/src/pages/PaymentConfirmation.tsx
    frontend/src/pages/BookingDetails.tsx

  Estado: CORRECTO — el precio siempre refleja las noches reales.

ETAPA 3 Y 4 — RBAC ACTIVADO + JOIN CORREGIDO [COMPLETADAS]
  Problema 1: authorize() middleware existia pero NUNCA se aplicaba en rutas.
  Cualquier usuario autenticado podia llamar endpoints de administrador.

  Problema 2: La query de reservaciones usaba un join incorrecto:
    .select('*, rooms(*), room_types:rooms(room_types(*))')
  El alias 'room_types:rooms(...)' es incorrecto y produce estructura inesperada.

  Solucion aplicada:
  - authorize() refactorizado: en vez de consultar la tabla 'profiles'
    (que podria no existir), lee user_metadata.role directamente del JWT.
    Esto elimina una query extra y hace el check instantaneo.
  - roomRoutes.ts: POST, PATCH y DELETE protegidos con authorize(['admin']).
  - reservationRoutes.ts: nuevo GET /admin/all protegido con authorize(['admin']).
  - reservationController.ts: nuevo getAllReservations() sin filtro de user_id.
  - Join corregido a: .select('*, rooms(*, room_types(*))') en todos los metodos.

  Archivos modificados:
    backend/src/middlewares/authMiddleware.ts
    backend/src/routes/roomRoutes.ts
    backend/src/routes/reservationRoutes.ts
    backend/src/controllers/reservationController.ts

  Estado: CORRECTO — RBAC activo. Solo admins acceden a rutas de administrador.

--------------------------------------------------------------------------------
2.1 CONFIGURACION DE AUTH — CRITICO
--------------------------------------------------------------------------------

  ACCION: Deshabilitar confirmacion de email para desarrollo.

  Por que: Supabase exige por defecto que el usuario confirme su email antes
  de poder hacer login. Esto hace que un usuario recien registrado NO pueda
  iniciar sesion inmediatamente — el login devuelve error.

  Donde: Authentication -> Providers -> Email
  Desactivar: "Confirm email" (toggle OFF)

  IMPORTANTE: Antes del despliegue en produccion (Sprint 5) volver a activarlo
  y configurar el template del email de confirmacion.

--------------------------------------------------------------------------------
2.2 CREAR UN USUARIO ADMINISTRADOR — CRITICO
--------------------------------------------------------------------------------

  Actualmente los 3 usuarios en la BD tienen role: 'guest'. Sin un usuario
  admin no se puede acceder al panel de administracion (/admin) ni usar
  los endpoints GET /reservations/admin/all, POST/PATCH/DELETE /rooms.

  OPCION A (recomendada) — Desde Supabase Dashboard:
    1. Authentication -> Users -> seleccionar tu usuario
    2. Edit -> Raw user meta data -> editar el JSON:
       {
         "full_name": "Tu Nombre",
         "role": "admin"
       }
    3. Save

  OPCION B — Registrar un nuevo usuario con rol admin:
    POST /api/auth/register
    {
      "email": "admin@hotelvertice.com",
      "password": "password123",
      "full_name": "Administrador",
      "role": "admin"
    }
    (funciona porque el register acepta el campo role del body)

2.4 ROW LEVEL SECURITY (RLS) — INFORMATIVO
--------------------------------------------------------------------------------

  El backend usa la SUPABASE_SERVICE_ROLE_KEY que bypasea RLS completamente.
  Por lo tanto las politicas RLS no afectan el funcionamiento actual.

  SIN EMBARGO, para el despliegue en produccion (Sprint 5) se recomienda:
  1. Habilitar RLS en todas las tablas.
  2. Crear politicas que permitan al service role hacer cualquier operacion.
  3. Bloquear acceso directo desde el frontend (la anon key no deberia
     poder leer/escribir reservas directamente).

  Esto no es urgente ahora, pero es buena practica antes de la demo final.

--------------------------------------------------------------------------------
2.5 TABLA PROFILES — NO REQUERIDA
--------------------------------------------------------------------------------

  El codigo original dependia de una tabla 'profiles' para leer roles.
  Esta dependencia fue ELIMINADA en Etapa 3 y 5. El rol ahora se lee
  directamente desde user_metadata del JWT de Supabase Auth.

  La tabla 'profiles' puede existir o no — el sistema ya no la usa.
  No es necesario crearla ni borrarla.
